## 1. 售后域

### 1.1 `public.after_sale_requests`

| 策略名 | 类型 | 覆盖动作 | 条件 |
|---|---|---|---|
| `Users can read own after sale requests` | `FOR SELECT` | 只读 | `auth.uid() = user_id` |

**意图**：用户只能查看自己的退款申请；所有写入路径走 Edge Function（service role）。

**为什么没有 INSERT / UPDATE / DELETE policy**：
- 创建走 `create-after-sale-request` Edge Function，内部校验订单归属、状态机合法性，并一次性写入 `requested_amount` / `approved_amount` / `snapshot`。
- 撤销走 `cancel-after-sale-request` Edge Function，统一套用状态机守卫。
- 审核 / 退款走后台运维 service role（当前 V1 为手动触发）。
- 如果给用户直接写权限，绕过 Edge Function 就会触发 `handle_after_sale_request_updated` 和 `handle_after_sale_request_order_sync` 两个 trigger，在绕过上面 3 条路径的情况下直接写入 `notifications` 与 `orders` 字段 — 风险面太大。

**已有防护**：
- `uq_after_sale_requests_active_order` 唯一索引保证同一订单同时只有一个处于处理中的申请。
- 两条 trigger（`trg_after_sale_request_updated_notification`、`trg_after_sale_request_order_sync`）在状态变更时同步通知与订单快照。
- Edge Function 层应校验 `auth.uid() = user_id`、`orders.user_id = auth.uid()`、状态机可达性。

**下次迭代要补**：
- Edge Function 层的调用速率限制（防止刷单发起大量售后）。
- 后台审核端需要一条独立策略或 service role endpoint，目前代码库里没有看到审核动作入口。

### 1.2 `public.after_sale_evidences`

| 策略名 | 类型 | 覆盖动作 | 条件 |
|---|---|---|---|
| `Users can read own after sale evidences` | `FOR SELECT` | 只读 | 关联 request 归属当前用户 |
| `Users can insert own after sale evidences` | `FOR INSERT` | 追加凭证 | `WITH CHECK` 校验关联 request 归属 |

**意图**：用户可以为自己的售后申请上传凭证；不可删除 / 修改历史凭证，保留完整审计链。

**潜在风险**：
- 没有 UPDATE / DELETE policy — 如果 V2 要允许用户撤回凭证，需要追加 policy，**并同步在 Edge Function 层记录操作日志**。
- `sort_order` 由客户端填入，恶意客户端可以乱序插入干扰展示 — 但不影响核心安全。

**已有防护**：
- `after_sale_evidences.request_id` 的外键 `ON DELETE CASCADE` 保证主申请被删时凭证一起清理。
- Storage bucket 的文件夹隔离（见下）。

---

### 1.3 Storage bucket `after-sale-evidences`

四条策略全部绑定 `bucket_id = 'after-sale-evidences' AND auth.uid()::text = (storage.foldername(name))[1]`：

| 策略名 | 覆盖动作 |
|---|---|
| `After sale evidences read` | `SELECT` |
| `After sale evidences upload` | `INSERT` |
| `After sale evidences update` | `UPDATE` |
| `After sale evidences delete` | `DELETE` |

**意图**：按 `{user_id}/{request_id}/{timestamp}.{ext}` 的路径前缀做文件夹隔离，用户只能看到自己文件夹下的内容。

**与 `src/lib/afterSale.ts` 的契约**：
- `uploadAfterSaleEvidence` 使用 `${userId}/${requestId}/${Date.now()}-${sortOrder}.${ext}` 的路径格式，完全匹配 `storage.foldername(name)[1] = auth.uid()` 的策略。
- `resolveEvidenceDisplayUrl` 使用签名 URL（1 小时有效）而不是 public URL，符合 bucket `public = false` 的设计。

**潜在风险**：
- 用户可以 UPDATE / DELETE 自己的历史凭证 — 与 1.2 的"追加写"设计不一致。如果想强化审计链，应收紧为"只允许 INSERT，禁止 UPDATE / DELETE"，或由 Edge Function 在接单时固化快照。

**下次迭代要补**：
- bucket 策略与表策略对齐：要么都放开更新删除，要么都关上。
- 凭证 MIME 类型白名单（当前仅在 `uploadAfterSaleEvidence` 里限制为 jpg/png，bucket 层没有约束）。

## 2. 推送域

### 2.1 `public.push_devices`

| 策略名 | 类型 | 覆盖动作 | 条件 |
|---|---|---|---|
| `Users can read own push devices` | `FOR SELECT` | 只读 | `auth.uid() = user_id` |
| `Users can manage own push devices` | `FOR ALL` | 增删改 | `auth.uid() = user_id`（`USING` + `WITH CHECK`） |

**意图**：用户可直接维护自己的推送设备；注册 / 注销也可以走 `register-push-device` / `unregister-push-device` Edge Function 做更严格的校验与 push token 归一。

**为什么同时保留 `FOR ALL`**：
- 支持客户端在网络波动时直接本地 `upsert` 最新设备状态（当前代码全部走 Edge Function，但 policy 留有后路）。

**潜在风险**：
- `uq_push_devices_expo_push_token` 的唯一约束是跨用户全局的 — 这是对的，Expo push token 全局唯一。如果恶意用户伪造别人的 token 尝试插入，会触发唯一约束冲突；但这不能完全防止绕过 Edge Function 的恶意入库（例如构造脏设备记录）。
- 因此 Edge Function 仍应是首选写入路径，本地直写仅在离线场景使用。

**下次迭代要补**：
- 客户端直写路径加日志审计，跟踪非 Edge Function 的写入频次。
- 考虑将 `FOR ALL` 收敛为 `INSERT` + `UPDATE`，禁止客户端 DELETE（删除设备应该走 Edge Function）。

---

### 2.2 `public.push_preferences`

| 策略名 | 类型 | 覆盖动作 | 条件 |
|---|---|---|---|
| `Users can read own push preferences` | `FOR SELECT` | 只读 | `auth.uid() = user_id` |
| `Users can manage own push preferences` | `FOR ALL` | 增删改 | `auth.uid() = user_id` |

**意图**：偏好（总开关 + 分类开关 + 免打扰时段）由用户直接维护，不需要 Edge Function 中转。

**已有防护**：
- `loadPushPreference` / `updatePushPreference` 使用 `upsert(..., { onConflict: "user_id" })`，与主键约束共同保证幂等。

**潜在风险**：
- 免打扰时段（`quiet_hours_start` / `quiet_hours_end`）目前没有被 `dispatch-push-queue` 函数消费 — 需要验证该函数是否读取了这两列并做时间窗口判断。

**下次迭代要补**：
- 在推送分发 Edge Function 中实际消费 `quiet_hours_*` 字段；补测试覆盖跨日时段（`23:00-07:00`）。

### 2.3 `public.push_dispatch_queue` / `public.push_delivery_logs`

**两张表都：**
- 启用了 RLS（`alter table ... enable row level security`）
- **未定义任何 policy**

**意图**：对 `authenticated` 用户完全关闭；只有 service role 可以读写。

**为什么这样设计**：
- 队列表与日志表是系统内部资产，包含投递状态、失败原因、设备 token 明文等敏感字段。
- 用户应通过自己的 `push_delivery_logs` 归属看结果，但目前没做这个 —— V1 就让用户看不到投递细节，防止泄露 token 与失败原因。

**已有防护**：
- `trg_notification_push_queue` trigger 仅在 `notifications` 写入时自动入队，`handle_notification_push_queue` 函数标注 `security definer`，即使用户有 INSERT 权限到 `notifications` 也不会绕过。
- `uq_push_dispatch_queue_notification_id` 唯一索引防止同一通知多次入队。

**潜在风险**：
- `resolve_push_type_from_notification` 函数 `security definer` — 暂时没有副作用（纯函数），但签名加了 `security definer` 需要审计后续是否被扩展。
- `dispatch-push-queue` Edge Function 必须使用 service role key；PUSH_DISPATCH_SECRET（`.env.example:62`）需保留。

**下次迭代要补**：
- 如果要给用户展示"推送是否送达"，需要给 `push_delivery_logs` 加一条 `FOR SELECT USING (...)` 的归属策略（通过 queue_id → notification_id → user_id 联查），**不能放开全表**。

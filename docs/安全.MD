# 项目安全审查报告

- 审查日期：2026-03-22
- 审查项目：`liji-tea-app`
- 审查范围：前端代码、依赖清单、本地环境配置、Supabase 客户端调用链路
- 审查方式：
  - 依赖漏洞扫描：`npm audit --json`
  - 代码检索：认证、订单、支付、存储、上传、搜索、实时订阅相关逻辑
  - 配置核对：`.env`、`.gitignore`、`app.json`、`package.json`

## 一、审查结论

当前项目**不能判定为“没有安全漏洞”**。

本次审查未发现已知依赖漏洞，也未发现硬编码的高敏感密钥泄露问题；但在业务安全和客户端信任边界上，发现了数个风险点，其中包含**高危业务逻辑漏洞**。

综合判断如下：

- 依赖安全：当前未发现已知漏洞
- 密钥管理：未发现 `service_role`、私钥等高危泄露
- 业务安全：存在高危问题
- 会话存储：存在中风险问题
- 查询构造：存在中低风险问题

## 三、主要安全问题

## 1. 高危：订单金额与支付结果完全由客户端控制

### 问题描述

结算页会在前端本地计算订单总价，并将以下数据直接传入创建订单逻辑：

- 商品单价 `unitPrice`
- 订单总额 `total`
- 支付方式 `paymentMethod`

随后这些值会被直接写入数据库。

支付页又会从路由参数中读取 `orderId`、`total` 等值，等待几秒后直接调用更新逻辑，把订单状态修改为 `paid`。

### 风险影响

这意味着攻击者如果能够修改客户端请求、调试前端状态、篡改路由参数或重放请求，就可能：

- 以被篡改后的低价创建订单
- 在未真实支付的情况下把订单状态改成已支付
- 绕过真实支付网关流程

如果该项目面向真实交易场景，这是**阻断上线级漏洞**。

### 相关位置

- `src/app/checkout.tsx`
- `src/app/payment.tsx`
- `src/stores/orderStore.ts`

### 修复建议

- 订单总价必须由服务端计算，前端只提交 `productId`、`quantity`、`addressId` 等最小必要参数
- 支付成功状态只能由服务端根据支付网关回调或签名校验后更新
- 禁止前端直接调用“改订单为 `paid`”的数据库写操作
- 对订单状态迁移做服务端白名单约束，只允许合法状态流转

## 2. 高危：订单读取与更新过度依赖前端参数，存在越权风险

### 问题描述

跟踪页从路由参数直接读取 `orderId` 并查询订单详情。  
订单更新逻辑也主要按 `id` 执行更新。

从仓库代码来看，客户端对“当前用户是否拥有该订单”的约束并不强，真正的安全性取决于 Supabase 线上数据库是否已经正确启用 RLS 与策略。

### 风险影响

如果线上 RLS 没有正确部署，或策略存在缺口，则可能出现：

- 任意已登录用户查看其他用户订单
- 任意已登录用户修改其他用户订单状态
- 通过遍历订单 ID 获取他人订单信息

这类问题属于典型的 **IDOR（不安全直接对象引用）** / 越权访问风险。

### 相关位置

- `src/app/tracking.tsx`
- `src/stores/orderStore.ts`

### 当前审查中的关键判断

仓库中**只看到了 RLS 设计文档**，但没有看到可执行的 migration / SQL 文件来证明这些策略已经实际落库。

也就是说：

- “设计上打算启用 RLS” 是成立的
- “当前部署环境已经可靠启用 RLS” 无法从本仓库证明

### 修复建议

- 将 Supabase 表结构、RLS、Storage policy 以 migration 文件提交到仓库
- 订单读取接口即使有 RLS，也建议显式带上用户边界约束，减少误配置风险
- 对所有用户私有表统一做“只允许本人读取/修改”的后端策略检查

## 3. 中风险：认证会话未使用系统级安全存储

### 问题描述

当前 Supabase 客户端配置为：

- `storage: localStorage`
- `persistSession: true`

并通过 `expo-sqlite/localStorage/install` 做本地持久化。

项目虽然安装了 `expo-secure-store`，但代码里并未实际用它来保存认证会话。

### 风险影响

相较于系统级 Keychain / Keystore，当前方案在以下场景中的安全性更弱：

- Root / 越狱设备
- 调试或备份环境
- 设备被物理接触后的本地数据提取

这不一定意味着立即可利用，但确实提高了 token 被读取的概率。

### 相关位置

- `src/lib/supabase.ts`
- `package.json`
- `app.json`

### 修复建议

- 为 Supabase Auth 提供 `expo-secure-store` 适配器
- 将 access token / refresh token 的持久化从 `localStorage` 切换为系统安全存储
- 明确区分“普通本地缓存”和“认证凭据存储”

## 4. 中低风险：搜索条件直接拼接查询表达式

### 问题描述

商品搜索将用户输入的 `query` 直接拼接进：

- `.or(...)`
- `ilike`

形成动态查询表达式。

### 风险影响

当前查询目标是公开商品数据，实际危害低于订单与支付问题；但这种写法仍存在以下隐患：

- 特殊字符导致查询异常
- 构造出超出预期的过滤条件
- 后续若该模式被复制到私有数据查询中，可能演变为真正的注入问题

### 相关位置

- `src/stores/productStore.ts`

### 修复建议

- 对搜索输入做字符转义或白名单过滤
- 避免直接拼接 PostgREST 表达式字符串
- 尽量封装统一的安全查询构造方法，防止后续复制扩散

## 四、需要补充确认的盲区

以下内容无法仅通过当前仓库代码完全确认：

### 1. 线上 Supabase 是否已真正部署 RLS

仓库内有详细的 RLS 规划文档，但没有发现明确的 migration / SQL 执行文件，因此无法验证：

- RLS 是否已经在线上数据库启用
- 策略是否与文档一致
- Storage bucket 是否配置了访问策略

### 2. 是否存在真实支付后端

当前仓库中的支付流程明显是前端模拟流程。  
如果线上实际上另有服务端支付回调，则本仓库无法反映其真实性和完整性。

## 五、风险等级汇总

- 高危：订单金额与支付状态由客户端决定
- 高危：订单读取/更新依赖前端参数，RLS 是否真正部署无法验证
- 中风险：认证会话未落到系统级安全存储
- 中低风险：搜索查询字符串直接拼接

## 六、修复优先级建议

### 第一优先级

- 下线前端直接改订单状态为 `paid` 的能力
- 将订单创建、金额计算、状态更新迁移到服务端
- 校验线上 Supabase 是否真正启用并验证 RLS

### 第二优先级

- 提交数据库 migration 和策略文件，保证权限配置可审计、可追踪、可复现
- 为所有用户私有资源统一补足后端访问边界

### 第三优先级

- 将认证凭据存储迁移到 `expo-secure-store`
- 清理直接拼接查询表达式的写法

## 七、最终结论

当前项目在“依赖漏洞”和“明文高危密钥泄露”两个维度上表现正常，  
但在“支付可信性”和“订单权限边界”两个核心业务安全维度上，**存在不可忽视的问题**。

因此，本次审查的最终结论是：

> 当前项目**不能认定为无安全漏洞**，且在真实交易场景下**不建议直接上线**。

在完成服务端价格校验、支付回调落库、订单权限闭环、会话安全存储改造之前，建议将该项目视为**功能可演示、但未达到生产安全要求**的状态。

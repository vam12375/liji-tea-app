# 订单列表到物流追踪闭环设计（Liji Tea App）

- 日期：2026-03-24
- 项目：liji-tea-app
- 当前阶段：设计评审通过稿
- 设计范围：订单列表 -> 物流追踪页面的业务闭环真实化
- 推荐方案：基于现有订单数据的状态驱动履约页

---

## 1. 背景

当前项目已经具备以下基础能力：

- 订单列表页可读取真实订单数据
- 物流追踪页可根据 `orderId` 拉取订单详情
- 订单状态已接入真实字段：`pending`、`paid`、`shipping`、`delivered`、`cancelled`
- 订单详情查询已包含 `order_items` 和 `address` 关联数据

但当前物流追踪页仍然停留在演示状态，核心问题如下：

1. 页面展示了固定的地图背景、固定快递公司和固定运单号。
2. 时间线使用硬编码节点和固定日期，与真实订单状态无关。
3. 起点、终点、包裹信息中仍混有假数据。
4. 待付款、待发货等订单虽然能进入物流页，但页面表达不准确。

这导致“订单列表 -> 物流追踪”虽然有路由跳转，但并没有形成可信的业务闭环。

---

## 2. 设计目标

本次设计的目标如下：

1. 让订单列表中的任意订单都能稳定进入物流追踪页。
2. 让物流追踪页只展示基于真实订单数据推导出的履约信息。
3. 使用现有订单字段生成“准真实”的履约时间线，不再展示固定 mock 物流数据。
4. 让 `pending`、`paid`、`shipping`、`delivered`、`cancelled` 五种订单状态都有统一且可解释的物流页表现。
5. 保留当前实时订阅能力，在订单状态变化时自动刷新页面。

---

## 3. 非目标

本次设计不包含以下内容：

- 接入第三方物流 API
- 新增 `logistics` 数据表
- 引入真实运单号、承运商、骑手电话或地图轨迹
- 改造支付链路
- 改造订单列表筛选逻辑
- 改造个人中心订单统计
- 新增新的全局 store 或服务层

---

## 4. 当前现状

### 4.1 已有数据能力

当前订单模型已经具备以下字段：

- `id`
- `status`
- `total`
- `delivery_type`
- `created_at`
- `updated_at`
- `paid_at`
- `notes`
- `gift_wrap`
- `address`
- `order_items`

这些字段已经足够支持一个状态驱动的履约页。

### 4.2 当前页面问题

当前 [tracking.tsx](E:/liji-tea-app/src/app/tracking.tsx) 的问题集中在：

1. 地图区域是静态背景，不对应真实地址。
2. 快递信息卡展示固定承运商与固定运单号。
3. 时间线是常量数组，和订单状态、时间无关。
4. 包裹摘要仍带有兜底的假商品名和固定图片。

这些内容会误导用户，必须替换为订单驱动的数据展示。

## 5. 方案选型

### 5.1 方案 A：状态驱动物流页

做法：

- 保留当前订单表结构
- 使用 `status / created_at / paid_at / updated_at / address / order_items` 生成履约视图
- 删除固定地图、固定快递信息和固定时间线

优点：

- 改动最小，符合 KISS
- 不新增数据库结构，符合 YAGNI
- 逻辑集中在物流页和少量订单 store 支撑代码中
- 后续如接真实物流 API，可在此基础上平滑升级

缺点：

- 不是第三方物流意义上的真实轨迹
- 发货时间与签收时间只能用当前订单字段近似表达

### 5.2 方案 B：新增 logistics 表

优点：

- 可以承载更细粒度的节点、运单号、承运商和轨迹

缺点：

- 当前属于过度设计
- 需要新增 migration、store、回写逻辑和维护成本

### 5.3 方案 C：接第三方物流 API

优点：

- 最终效果最真实

缺点：

- 当前项目没有现成接口、凭证和服务端接入条件
- 范围远超本轮修复目标

### 5.4 结论

推荐采用 **方案 A：状态驱动物流页**。

这是当前项目下最小、最稳、最符合现有数据结构的方案。

## 6. 页面职责与边界

### 6.1 订单列表页职责

[orders.tsx](E:/liji-tea-app/src/app/orders.tsx) 保持当前职责：

- 拉取当前用户订单列表
- 提供筛选标签
- 点击订单后跳转到物流追踪页

本次不在订单列表页中加入新的业务判断或复杂条件。

### 6.2 物流追踪页职责

[tracking.tsx](E:/liji-tea-app/src/app/tracking.tsx) 改造为纯“订单履约视图”，负责：

- 根据 `orderId` 拉取当前订单详情
- 根据订单状态生成履约时间线
- 展示订单基础信息、收货信息和包裹摘要
- 根据订单状态更新页面主状态卡
- 在实时订阅推送后刷新当前视图

不负责：

- 计算订单金额
- 修改订单状态
- 接入第三方物流查询
- 伪造快递信息

## 7. 履约状态设计

物流页不再将自己视为“快递追踪页”，而是视为“订单履约进度页”。

### 7.1 状态映射

| 订单状态 | 页面主状态 | 页面说明 |
| --- | --- | --- |
| `pending` | 待支付 | 订单已创建，请尽快完成支付 |
| `paid` | 待发货 | 支付成功，仓库正在准备发货 |
| `shipping` | 运输中 | 包裹已发出，正在运输途中 |
| `delivered` | 已送达 | 包裹已送达，请注意查收 |
| `cancelled` | 已取消 | 订单已取消，本次履约已结束 |

### 7.2 页面统一入口

所有订单状态都允许进入物流追踪页。

这样做的原因：

- 路由入口统一，用户心智简单
- 待付款与待发货也是履约过程的一部分
- 避免在订单列表页分叉跳转逻辑

---

## 8. 时间线设计

### 8.1 时间线节点

标准履约节点定义如下：

1. `订单已创建`
2. `待支付` 或 `支付已完成`
3. `待发货`
4. `运输中`
5. `已送达`
6. `订单已取消`（仅取消单出现）

### 8.2 节点时间来源

- `订单已创建`：使用 `created_at`
- `支付已完成`：优先使用 `paid_at`，没有则回退到 `updated_at`
- `待发货`：使用 `paid_at` 或 `updated_at`
- `运输中`：使用 `updated_at`
- `已送达`：使用 `updated_at`
- `订单已取消`：使用 `updated_at`

### 8.3 状态到时间线的映射规则

#### `pending`

- `订单已创建`：已完成
- `待支付`：当前节点
- `待发货`、`运输中`、`已送达`：未完成

#### `paid`

- `订单已创建`：已完成
- `支付已完成`：已完成
- `待发货`：当前节点
- `运输中`、`已送达`：未完成

#### `shipping`

- `订单已创建`：已完成
- `支付已完成`：已完成
- `待发货`：已完成
- `运输中`：当前节点
- `已送达`：未完成

#### `delivered`

- `订单已创建`：已完成
- `支付已完成`：已完成
- `待发货`：已完成
- `运输中`：已完成
- `已送达`：已完成

#### `cancelled`

- `订单已创建`：已完成
- 若存在支付迹象，则保留 `支付已完成`
- 末尾追加 `订单已取消` 作为终止节点

### 8.4 时间展示规则

- 有真实时间：格式化为 `YYYY-MM-DD HH:mm`
- 无可靠时间：显示 `待更新`
- 禁止展示固定伪造日期和固定时间节点

## 9. 页面布局设计

物流追踪页统一由四个模块组成。

### 9.1 顶部状态卡

内容：

- 当前履约状态标题
- 当前状态解释文案
- 订单号后 8 位
- 下单时间
- 配送方式文案

作用：

- 第一时间告诉用户订单走到哪一步

### 9.2 收货信息卡

内容：

- 收货人姓名
- 脱敏手机号
- 完整收货地址

缺省处理：

- 没有关联地址时，显示 `暂未获取收货地址`

### 9.3 物流时间线卡

内容：

- 基于订单状态生成的履约节点
- 每个节点展示标题、说明、时间

视觉规则：

- 已完成节点：勾选或高亮实心圆
- 当前节点：重点高亮
- 未完成节点：弱化
- 取消节点：错误色处理

### 9.4 包裹摘要卡

内容：

- 商品首图 1 到 2 张
- 商品名称摘要
- 商品总件数
- 礼盒包装标记
- 订单备注

规则：

- 商品名称过长时不整串展示
- 优先展示前 1 到 2 个商品名，其余汇总为“等 N 件商品”
- 若无图片，使用图标占位，不再伪造外部图片

## 10. 数据流设计

### 10.1 页面数据流

1. 物流页读取 `orderId`
2. 页面挂载后调用 `fetchOrderById(orderId)`
3. 从 `useOrderStore` 读取 `currentOrder`
4. 页面内部根据 `currentOrder` 派生：
   - 主状态卡
   - 收货信息
   - 时间线
   - 包裹摘要
5. 若订阅收到订单状态更新，则刷新当前展示

### 10.2 store 责任划分

[orderStore.ts](E:/liji-tea-app/src/stores/orderStore.ts) 继续作为订单数据单一来源，只负责：

- 查询订单
- 缓存订单详情
- 响应实时更新

物流页本身只做展示派生，不引入新的全局 store。

## 11. 辅助函数设计

建议在 [tracking.tsx](E:/liji-tea-app/src/app/tracking.tsx) 内部保留以下最小辅助函数：

- `formatDateTime(value)`：格式化时间
- `maskPhone(phone)`：脱敏手机号
- `getTrackingStatusMeta(status)`：生成主状态卡文案、颜色、图标
- `buildTrackingTimeline(order)`：生成履约时间线
- `summarizeOrderItems(items)`：生成包裹摘要
- `getDeliveryLabel(deliveryType)`：将内部配送方式字段转为用户文案

不建议提前抽出独立 service 或新文件，以免过度设计。

---

## 12. 空态与异常态设计

### 12.1 加载中

- 展示页面级 loading
- 避免显示上一笔订单残留内容

### 12.2 订单不存在或参数无效

- 展示空态：`未找到订单信息`
- 提供返回订单列表入口

### 12.3 地址缺失

- 不伪造地址
- 展示 `暂未获取收货地址`

### 12.4 商品图片缺失

- 使用图标或纯色占位
- 不使用固定网络图片充数

### 12.5 已取消订单

- 顶部状态卡明确标红
- 时间线终止于 `订单已取消`

### 12.6 实时更新

- 保留现有订单订阅逻辑
- 当订单状态从 `paid -> shipping` 或 `shipping -> delivered` 变化时，页面自动刷新

## 14. 验收标准

满足以下条件，视为本轮闭环完成：

1. 从订单列表点击任意状态订单，都能进入物流页。
2. 物流页不再展示固定假物流数据。
3. 物流页能显示真实订单号、时间、配送方式、收货信息和包裹摘要。
4. 时间线会随订单状态变化而变化。
5. 页面具备加载中、空态、取消态等基本状态处理。
6. 实时更新后，顶部状态和时间线能同步刷新。

## 15. 风险与约束

### 15.1 已知约束

- 当前数据库没有真实物流字段
- 没有独立的发货时间和签收时间字段
- 没有运单号、承运商、快递轨迹数据

### 15.2 风险说明

- `shipping` 与 `delivered` 的具体时间只能基于 `updated_at` 近似表达
- 部分历史订单可能没有完整地址或商品图片
- 如果 `currentOrder` 不及时清理，页面可能短暂显示旧数据

### 15.3 风险结论

这些风险不会阻断本轮目标。  
本轮目标是先把“演示物流页”升级为“订单状态驱动的履约页”，而不是一次性做成真实物流系统。

---

## 16. 最终结论

本轮最合理的改造路径是：

- 保留订单列表现有入口
- 将物流页改造成状态驱动的履约页
- 用现有订单数据生成准真实履约时间线
- 删除会误导用户的固定 mock 物流内容
- 只做物流页和订单 store 的最小必要改动

这样可以在不新增后端结构、不依赖外部物流接口的前提下，最快把“订单列表 -> 物流追踪”这条链路补成一个可用、可信、可继续演进的业务闭环。
